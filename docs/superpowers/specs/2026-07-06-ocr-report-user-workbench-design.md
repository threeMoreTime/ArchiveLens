# 约字检索报告用户工作台重构设计

日期：2026-07-06

## 1. 目标

把现有的单文件离线 HTML 报告重构为一个以用户使用为中心的校对工作台。

这次设计的核心目标按优先级排序如下：

1. 快速回到命中结果的来源出处并完成核对
2. 在核对过程中支持按文档和页码做轻量筛选与研究浏览
3. 保留基础的快速查找能力，但不让它主导页面结构

本次重构采用“两步走”策略：

1. 先把当前 DJVU 阶段版报告重构到可用
2. 再为后续最终总报告保留可复用的界面模板

## 2. 设计原则

### 2.1 用户视角优先

页面只围绕用户动作组织：

- 看结果
- 找出处
- 做校对
- 留备注
- 继续下一条

不在主界面展示任何面向实现或调试的技术说明。

### 2.2 一屏完成主流程

用户的主流程不应在“表格页”和“查看弹层”之间来回切换。

因此主界面采用：

- 左侧结果清单
- 右侧固定详情区
- 详情区内直接完成校对动作

### 2.3 默认简单，必要时展开

用户一打开页面应能立即上手，因此默认只显示最必要的信息。

次要信息采用“更多信息”折叠区承载，默认收起。

### 2.4 连续处理优先

页面应服务连续校对，而不是一次性查看单条结果。

因此需要始终保留：

- 上一条
- 下一条
- 下一条待处理

## 3. 目标用户与典型场景

### 3.1 目标用户

- 正在人工核对 OCR 命中结果的研究者或整理者
- 需要快速确认“这个字到底出自哪本书、哪一页、什么上下文”的用户
- 需要在多条结果之间连续作出判断并记录结论的用户

### 3.2 核心场景

#### 场景 A：连续校对

用户打开报告后：

1. 先筛选文档和页码范围
2. 在左侧挑选一条结果
3. 在右侧同时查看出处页和截取小图
4. 结合上下文作出判断
5. 标记结论、填写备注
6. 直接进入下一条待处理结果

#### 场景 B：按文档查出处

用户知道要从某本书中找命中结果：

1. 先按文档筛选
2. 再按页码范围缩小范围
3. 在左侧浏览结果
4. 点开后立即回到右侧出处页进行核对

#### 场景 C：阶段性整理

用户完成一批校对后：

1. 页面自动保留本地进度
2. 用户可导出完整校对记录
3. 后续再次打开同一报告时可继续工作

## 4. 页面结构

整个页面分为四个层级。

### 4.1 顶部使用区

顶部不是“技术报告说明”，而是一个短而直接的使用区。

内容包括：

- 文档筛选
- 页码范围
- 只看有出处图片
- 只看待处理
- 清空筛选
- 导出校对记录

这个区域的职责是让用户快速进入目标范围，而不是承担说明书功能。

### 4.2 左侧结果清单

左侧只负责帮助用户快速决定“先点哪一条”。

每条结果仅展示：

- 文档名
- 页码
- 本页第几处
- 上下文短句
- 当前校对结论
- 是否有出处页
- 是否有截取小图

不在清单中展示完整路径、坐标、识别方式等内容。

### 4.3 右侧固定详情区

右侧是主工作区，用户点击任意结果后，详情立即同步更新。

详情展示顺序固定为：

1. 出处页
2. 截取小图
3. 上下文
4. 校对操作
5. 更多信息

### 4.4 底部或局部连续处理区

右侧详情区中必须始终保留连续处理能力：

- 上一条
- 下一条
- 下一条待处理

这部分要尽量就近放在校对动作区，减少来回移动。

## 5. 左侧结果清单设计

### 5.1 清单目标

清单不是总览数据库，而是浏览入口。

它必须做到：

- 让用户一眼看懂每条结果的核心信息
- 帮助用户快速找到值得优先处理的条目
- 在不展开详情的情况下知道当前结论状态

### 5.2 每条结果的信息密度

推荐结构：

- 第一行：文档名 + 第几页 + 第几处
- 第二行：上下文短句
- 第三行：校对结论 + 图片证据标记

### 5.3 状态表达

结论全部使用白话表达：

- 已确认
- 待判断
- 排除

不要直接把英文状态暴露给用户。

### 5.4 列表排序

默认排序规则：

- 按文档顺序
- 再按页码顺序
- 再按本页出现顺序

这是为了贴合连续翻书与连续核对的实际习惯。

### 5.5 列表切换

清单顶部提供轻量切换：

- 全部结果
- 只看待处理
- 只看已确认

其中“只看待处理”是最重要的工作视图。

## 6. 右侧详情区设计

### 6.1 顶部摘要

右侧顶部只保留当前条目的最短摘要：

- 文档名
- 页码
- 本页第几处

用户不需要先读解释，进入详情就该直接核对。

### 6.2 出处页

出处页是右侧第一优先级信息。

要求：

- 默认直接显示整页
- 当前命中位置需要明显高亮
- 用户可以在当前页面内直接定位
- 同时保留“单独查看出处页”的入口

这是整个页面中最关键的证据区。

### 6.3 截取小图

截取小图是第二优先级信息。

要求：

- 独立显示
- 不埋在弹层或长说明文字中
- 默认就能看见

它的作用是让用户快速判断字形，而不必先自己放大整页。

### 6.4 上下文

上下文放在图片之后，用于语义辅助判断。

上下文展示应尽量简洁，不需要在主视图塞入过长文本。

### 6.5 校对动作区

校对动作区必须与详情内容同屏出现。

用户应能直接完成：

- 标记已确认
- 标记待判断
- 标记排除
- 输入备注
- 保存当前进度
- 导出校对记录

### 6.6 更多信息

更多信息默认收起。

这里仅放少量辅助内容，不参与主流程。

技术信息不消失，但不能抢占主界面层级。

## 7. 顶部筛选与浏览控制

### 7.1 常驻筛选项

顶部常驻筛选只保留以下项目：

- 按文档
- 页码范围
- 只看有出处图片

此外补一个轻量工作开关：

- 只看待处理

### 7.2 不进入主区的筛选项

本轮不把大量高级条件放入顶部主区域。

原因：

- 会拖慢首次上手
- 会让主界面更像后台工具而不是工作台
- 用户当前主目标不是做复杂检索，而是快速溯源和校对

### 7.3 清空与恢复

用户必须可以快速：

- 清空筛选
- 恢复上次工作现场

## 8. 保存与导出

### 8.1 本地继续

页面应自动保留以下内容，供下次继续：

- 当前筛选条件
- 当前查看到的条目
- 已做出的校对结论
- 已填写的备注

### 8.2 导出能力

页面需要支持导出完整校对记录。

导出目标同时满足两类用途：

1. 留痕归档
2. 后续二次整理或分析

因此导出应是完整记录，而不是仅导出最终状态摘要。

### 8.3 导出入口

导出入口应放在用户易于理解和易于找到的位置，建议位于顶部使用区右侧或校对动作区附近。

导出名称直接使用：

- 导出校对记录

## 9. 文案策略

### 9.1 主界面白话化

主界面所有关键文案统一走白话表达。

推荐替换示例：

- 原页定位 -> 出处页
- 裁剪图 -> 截取小图
- verification_status -> 校对结论
- confirmed -> 已确认
- needs_review -> 待判断
- rejected -> 排除

### 9.2 技术信息处理

技术信息采用“默认隐藏，必要时展开”的策略。

因此：

- 主界面不出现技术介绍
- 仅在“更多信息”中保留少量辅助信息
- 不使用实现视角或调试视角的文案

## 10. 范围边界

### 10.1 本轮在范围内

- 重构当前 DJVU 阶段版报告的展示结构
- 把主流程改造成“列表 + 固定详情 + 校对动作”一体化
- 统一主界面文案为白话表达
- 支持本地继续与导出校对记录
- 为最终总报告保留可复用结构

### 10.2 本轮不在范围内

- 把页面做成技术审计台
- 在主界面展示大量技术字段
- 以复杂高级检索代替简单上手
- 先实现本轮未确认、但可能以后才会需要的扩展能力

## 11. 验收标准

如果重构成功，用户应能在不看说明的前提下完成以下动作：

1. 打开页面后快速按文档与页码缩小范围
2. 在左侧快速选中一条结果
3. 在右侧立即看到出处页和截取小图
4. 结合上下文完成判断
5. 记录结论与备注
6. 直接继续下一条待处理结果
7. 关闭页面后下次还能继续
8. 在需要时导出完整校对记录

## 12. Loading 反馈设计

### 12.1 总体策略

本轮采用混合式 loading 策略：

- 结果切换时使用轻提示
- 首次打开和筛选刷新时使用骨架屏
- 导出时只让按钮进入短暂工作中状态

目标不是制造“很重的等待感”，而是明确告诉用户页面正在处理，避免误以为点击无效或内容卡住。

### 12.2 结果清单切换

当用户从左侧结果清单切换到另一条结果时：

- 左侧当前选中项立即切换
- 右侧详情区进入轻 loading 状态
- loading 采用半透明遮罩加小型进度提示
- 遮罩覆盖范围仅限右侧详情区

这一阶段不建议用整块骨架替换详情内容，否则会打断连续校对节奏。

### 12.3 顶部筛选刷新

当用户修改文档、页码范围或“只看待处理”等筛选条件时：

- 左侧结果清单先显示骨架占位
- 右侧详情区进入轻 loading 状态
- 待筛选完成后再同步更新左右两栏

这是因为筛选刷新不仅改变当前条目，也可能改变整组结果，用户更需要明确知道列表正在重新整理。

### 12.4 首次打开与恢复现场

当页面首次打开，或从本地恢复上次进度时：

- 左侧结果清单显示骨架屏
- 右侧详情区显示骨架屏
- 等首轮数据和图片准备好后再切换成真实内容

这样可以避免大文件初次渲染时的短暂空白或突兀跳变。

### 12.5 导出校对记录

点击“导出校对记录”后：

- 按钮立即进入工作中状态
- 按钮文案临时改成“正在导出”
- 导出完成后恢复原文案

导出过程不需要整页 loading，因为它不影响当前浏览与校对。

### 12.6 本轮不加 loading 的位置

以下操作本轮不建议额外加 loading：

- 修改校对结论
- 输入备注
- 本地自动保存

原因是这些操作目前都在本地即时完成，若强行加入 loading，反而会制造额外迟滞感和心理负担。

### 12.7 视觉原则

loading 表现应满足以下要求：

- 不遮住顶部筛选入口
- 不让页面整体闪烁
- 不让已有内容完全消失
- 能让用户一眼看懂“正在处理哪里”

换句话说，loading 只标记变化区域，不重置整个工作台。

## 13. 风险与后续

### 13.1 主要风险

- 如果页面继续保留过重的信息密度，主流程会再次被技术细节压住
- 如果右侧详情区仍依赖弹层，连续校对效率会明显下降
- 如果保存与导出入口过深，用户会对“是否真的保存了”缺乏确定感
- 如果 loading 做得过重，会破坏连续校对节奏
- 如果 loading 覆盖范围过大，用户会误以为整个页面失去响应

### 13.2 后续衔接

本设计完成后，下一步应输出实施计划，拆分为：

- 页面结构调整
- 主视图文案重命名
- 左侧清单重排
- 右侧详情改造
- 本地保存与导出接入
- loading 状态与骨架屏接入
- 验证与回归检查
